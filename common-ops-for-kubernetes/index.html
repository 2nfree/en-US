Rollout and Rollback

Changes to the Pod Template: If there are updates to the .spec.template, such as changes to labels, annotations, or the image (e.g., .spec.template.spec.containers.image), a new rollout will be triggered. This creates a new ReplicaSet to replace the old one.
Other changes won’t trigger a rollout: For instance, scaling the Deployment (changing the number of replicas via .spec.replicas) does not trigger a rollout, because it doesn’t modify the Pod template. Instead, the existing ReplicaSet will simply add or remove Pods to match the desired number of replicas.

StatefulSet and DaemonSet can also use the above commands, you just need to replace the command keywords.

Resource Quotas

Request and Limit (workload or pod)

K8S uses two abstract concepts, Request and Limit, to support resource scaling and quota management. Kubernetes manages resources for the following two aspects:

  • Request: Primarily used for flexible configuration of resource usage when deploying an application. Kubernetes uses this value to decide which node to schedule the container on.
  • Limit: Primarily used to enforce resource limits on application containers, defining the maximum resource quota a container can consume.

QoS (Quality of Service)

When the cluster resources are insufficient, Kubernetes makes eviction decisions based on the QoS class of the Pods to free up resources.

  • When Request = Limit, the QoS class is Guaranteed, and the Pod will only be killed if resource usage exceeds the limit.
  • When Request < Limit (including workloads or Pods that only set a Request but no Limit), the QoS class is Burstable. These Pods may be killed when node resources are insufficient.
  • When neither Request nor Limit is set, the QoS class is BestEffort. These Pods are the first to be killed if node resources are insufficient for scheduling.

ResourceQuota/LimitRange (namespace)

ResourceQuota/LimitRange is used to control the resource usage in a specific namespace, ultimately achieving fair use of the cluster and cost control.

  • Limits the computing resource usage of running Pods.
  • Restricts the number of persistent storage volumes to control access to storage.
  • Limits the number of load balancers to control costs.
  • Prevents abuse of network ports.
  • Provides default compute resource requests to help the system make more optimized scheduling decisions.
